1. Executive Summary
BREVIO V9.1 is a forward-only addendum to the V9 Consolidated Master Blueprint. It closes 13 gaps identified through a systematic feature audit against a comprehensive AI executive assistant capability model. V9.1 does not modify any existing V9 schema, workflow, or policy. It adds new tables, new workflows, new API endpoints, and new policy rules that integrate with the existing V9 architecture.
V9 excels at the hard engineering layer: deterministic execution, security, provisioning, and observability. V9.1 adds the soft intelligence layer: the features that make BREVIO feel like it truly knows the operator, grows with them over time, and proactively works toward their goals.

3.3 Trust-Building Autonomy (GAP-11)
Autonomy levels in V9 are static per-domain settings. V9.1 adds a trust scoring system that tracks the AI's track record per domain and can propose autonomy promotions when confidence thresholds are met. Promotions always require operator notification (NNR-V91-007). The trust score formula is deterministic and auditable.

3.3.1 autonomy_trust_scores
UNIQUE constraint: (workspace_id, domain). Trust score formula: trust_score = (success_count_30d - 2*failure_count_30d - 3*override_count_30d) / GREATEST(success_count_30d + failure_count_30d + override_count_30d, 1). Promotion threshold: trust_score >= 0.85 AND success_count_30d >= 20 AND failure_count_30d = 0 for trailing 14 days.

3.5 Daily Introspection and Log Capture (GAP-09, GAP-13)
V9 defines the memory_type enum value 'daily_log' but has no workflow to populate it. V9.1 adds a daily capture system with configurable templates and an introspection workflow that the AI runs at the end of each session (or on a scheduled cadence). Captures include: what was accomplished, what was learned, what's pending, and what the AI would do differently.

3.5.1 introspection_templates
Default template sections: accomplishments (what was completed today), learnings (new information or patterns discovered), pending_items (what remains open and needs attention), friction_noted (what was harder than expected and why), operator_mood_signal (observed operator energy/stress level if detectable), improvement_notes (what the AI would do differently next time).

3.5.2 daily_captures
UNIQUE constraint: (workspace_id, capture_date). Each capture creates a corresponding memory_item with type='daily_log' and embeds a summary for semantic retrieval.

3.6.4 debt_resolution_tasks
Debt resolution tasks are generated during codebase ingestion or cross-repo analysis. They can be linked to goals and milestones so that resolving technical debt contributes to goal progress. Tasks have dependency ordering to prevent out-of-sequence remediation. The linked_milestone_id allows a debt task completion to automatically advance a goal milestone.

3.8 Project Templates and Code Context Export (GAP-03, GAP-04)
3.8.1 project_templates
The components_json field contains an ordered array of reusable building blocks extracted from source repos. When a template is instantiated, each component is rendered in array order with variables substituted. This eliminates the need for a separate components table while preserving the ability to mix and match atomic pieces like CI configs, linting setups, Docker configurations, and directory scaffolds.

3.8.2 code_context_exports
The code context export assembles: repo profile, architecture patterns, coding conventions, known technical debt, dependency graph, tribal knowledge from memory, and active goals related to the repo. Output is formatted for consumption by external AI tools (GitHub Copilot, Claude Code, Cursor, etc.).

3.9 Self-Modification Controls (GAP-07)
V9 has no policy governing whether BREVIO can modify its own configuration (workspace_personas, workspace_behavior_policies, user_domain_autonomy_settings, learning_config, etc.). V9.1 adds explicit self-modification policies that default to DENY (NNR-V91-004).

3.9.1 self_modification_policies
ExecutionGate enforcement: when a tool call targets any of the above tables and the actor is the AI (not the operator directly), the gate checks self_modification_policies for that workspace. If the action is 'deny', the call is blocked with reason SELF_MODIFICATION_DENIED. If 'require_approval', an approval request is created. If 'allow_with_audit', the call proceeds but an audit_log_entry is emitted with event='BREVIO.self_modification.executed.v1'.

13. Migration
V9.1 is a forward-only migration addendum. It applies as migration 002_BREVIO_v91_soft_intelligence.sql after the V9 migration 001_BREVIO_v9_init.sql. No V9 tables are modified. All new tables follow V9 conventions: UUIDv7 PKs, workspace-scoped RLS, pgcrypto and pgvector extensions.

The migration creates all new enum types first, then all new tables in dependency order, then all RLS policies, then all indexes.

14. Closure Statement

BREVIO V9.1 closes all 13 gaps identified in the feature audit. It adds 23 new tables, 21 new enums, 8 new Temporal workflows, 32 new API endpoints, 8 new OPA policy rules, 13 new JSON schemas, 8 new prompt registry entries, 25 new metrics, 31 new canonical events, and 11 new acceptance gates.

V9.1 is purely additive. It does not modify any V9 table, enum, workflow, endpoint, policy, schema, prompt, metric, or event. All V9 acceptance gates, non-negotiable rules, and closure checks remain in force. The 24 new tables integrate cleanly with the existing V9 schema through standard foreign key relationships to workspaces, code_repositories, memory_items, ingress_turns, and other V9 tables.

V9.1 transforms BREVIO from a powerful execution engine into an AI executive assistant that learns the operator's goals, tracks progress toward them, earns greater autonomy through demonstrated competence, captures institutional knowledge through daily reflection, detects patterns across codebases, and proactively recommends new capabilities. It bridges the gap between infrastructure excellence and personal intelligence.
