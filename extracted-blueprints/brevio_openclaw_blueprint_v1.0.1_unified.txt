2.1.1 Architecture
The Plugin Hook System implements a composable event interception layer across every significant stage of Brevio's Brain plane agent loop. Hooks are synchronous or asynchronous registered functions (plugins) that receive a context envelope, can mutate it, emit side effects, or return HALT to abort the current pipeline stage.
Hook lifecycle: register → persist → evaluate at runtime → dispatch ordered callbacks → collect mutations → continue or halt.
Hook Points (complete enumeration)

Hook Registration Contract
interface HookRegistration {
  id:         string;   // UUIDv7, assigned at registration
  hook_point: HookPoint;
  plugin_id:  string;   // must reference plugins.id
  priority:   number;   // 0-1000, lower = earlier execution
  is_async:   boolean;  // if true, run in parallel; collect mutations at barrier
  timeout_ms: number;   // 0 = no timeout (sync only); max 30000
  can_halt:   boolean;  // must be true to return HALT signal
  config:     Record<string, unknown>; // static plugin config, JSON
  enabled:    boolean;
  created_at: string;   // ISO-8601
}

Hook Dispatch Algorithm
Hooks for a given point are dispatched in priority order. Synchronous hooks execute sequentially; async hooks are dispatched in parallel with a barrier at timeout_ms. Mutations are merged using last-write-wins by priority. If any hook returns HALT and can_halt=true, the pipeline is aborted and the halt_reason is logged.
function dispatchHooks(hookPoint, context):
  registrations = sortByPriority(getEnabled(hookPoint))
  sync_hooks  = filter(registrations, r => !r.is_async)
  async_hooks = filter(registrations, r => r.is_async)
 
  // Execute sync hooks sequentially
  for hook in sync_hooks:
    result = executeWithTimeout(hook, context, hook.timeout_ms)
    if result.signal == HALT and hook.can_halt:
      logHalt(hookPoint, hook.id, result.halt_reason)
      return HaltResult(result.halt_reason)
    mergeMutations(context, result.mutations, hook.priority)
 
  // Execute async hooks in parallel with barrier
  results = awaitAll(async_hooks, context, max_timeout=30000)
  for result in sortByPriority(results):
    if result.signal == HALT and hook.can_halt:
      logHalt(hookPoint, hook.id, result.halt_reason)
      return HaltResult(result.halt_reason)
    mergeMutations(context, result.mutations, hook.priority)
 
  return ContinueResult(context)


2.2.1 Architecture
All inbound DM (direct message) senders are untrusted by default. The Gateway executes a pairing state machine that issues a short-lived numeric pairing code to unknown senders and only promotes them to the allowlist after explicit operator approval. The dmPolicy has two modes: pairing (default, secure) and open (explicit opt-in, logged).
DM Pairing State Machine
States:  UNKNOWN → PENDING_PAIRING → APPROVED | REJECTED | EXPIRED
 
UNKNOWN:
  on inbound_message:
    if sender in allowlist → APPROVED (skip pairing)
    if dmPolicy == OPEN    → APPROVED (log open-mode grant)
    else:
      generate pairing_code = crypto.randomInt(100000, 999999)
      create pairing_request(sender, code, expires_at = now + 15min)
      send pairing_invite_message(sender, code)
      transition → PENDING_PAIRING
 
PENDING_PAIRING:
  on approve_command(sender, code):
    if code matches and not expired → add to allowlist → transition → APPROVED
    if expired                      → transition → EXPIRED (resend allowed)
    if wrong code (3x)              → transition → REJECTED (24h cooldown)
  on timeout(15min):
    transition → EXPIRED
 
APPROVED:
  on revoke_command(sender):
    remove from allowlist → transition → UNKNOWN
 
REJECTED:
  on cooldown_elapsed(24h):
    transition → UNKNOWN

2.3.1 Architecture
The Session Tool Protocol provides a minimal, deterministic agent-to-agent (A2A) coordination model. Four tool primitives are exposed to agents: sessions_list, sessions_history, sessions_send, sessions_spawn. The protocol enforces a ping-pong alternation loop with explicit REPLY_SKIP and ANNOUNCE_SKIP escape tokens.
A2A Tool Primitives

Ping-Pong Loop Protocol
Protocol: Requester (R) and Target (T) alternate turns.
 
R sends sessions_send(target_id, message, idempotency_key)
  → T receives turn, executes, replies via sessions_send(requester_id, response)
  → R receives response, processes
  → R may send next message OR emit REPLY_SKIP to terminate loop
  → T receives REPLY_SKIP → closes coordination, returns final_result
 
Invariants:
  - Max loop depth:  configurable, default 10 turns per coordination
  - Turn timeout:    30 seconds per turn; expired turns emit REPLY_SKIP automatically
  - Idempotency:     each sessions_send requires unique idempotency_key
  - Loop detection:  same message hash within session → auto REPLY_SKIP


2.4.1 Architecture
Each channel session has a dedicated queue lane that serializes agent runs and enforces an inbound message handling policy during an active run. Three queue modes govern how new messages arriving during an active run are processed.
Queue Lane Modes

Lane Lifecycle
On session_start:
  create lane_record(session_id, mode=collect, status=IDLE)
  create Redis queue: brevio:lane:{session_id}
 
On inbound_message(session_id, message):
  lane = getLane(session_id)
  if lane.status == IDLE:
    lane.status = RUNNING
    enqueue(message) → Temporal workflow
  else:  // lane is RUNNING
    if lane.mode == COLLECT:  push to buffer list
    if lane.mode == STEER:    replace steer_context with message
    if lane.mode == FOLLOWUP: push to followup_queue
 
On run_complete(session_id):
  lane.status = IDLE
  if buffer_queue non-empty:
    flush_buffer → new run

2.6.1 Architecture
Skills (prompt context files representing tool/domain capabilities) are loaded at session start only when their prerequisite conditions are satisfied. This prevents token bloat from injecting all 153 skills unconditionally and ensures context relevance.
Frontmatter Gate Schema
# skill: calendar_integration.md
metadata:
  openclaw.always: false       # if true, bypasses all gate checks
  scope: per-agent             # per-agent | shared
  version: 1.0.0
requires:
  bins:
    - gcal-cli                 # must be in PATH
  env:
    - GOOGLE_OAUTH_CLIENT_ID   # must be non-empty
    - GOOGLE_OAUTH_CLIENT_SECRET
  config:
    - calendar.enabled         # must be true in user config
    - calendar.oauth_token     # must be non-null

Gate Evaluation Logic
function evaluateSkillGate(skill, session):
  if skill.metadata.always == true: return PASS
 
  for bin in skill.requires.bins:
    if not which(bin): return FAIL(reason=f"binary {bin} not in PATH")
 
  for envVar in skill.requires.env:
    if not os.getenv(envVar): return FAIL(reason=f"env {envVar} missing")
 
  for configKey in skill.requires.config:
    val = session.user_config.get(configKey)
    if val is None or val == false: return FAIL(reason=f"config {configKey} not set")
 
  return PASS
 
// Skills resolved ONCE at session_start.
// Hot-reload: file watcher triggers re-evaluation on skill file change.


2.7.1 Architecture
All side-effecting RPC calls from Gateway into Brain (and Brain into Hands) must carry a client-generated idempotency_key. The server maintains a dedupe cache with a TTL of 24 hours. Duplicate requests within TTL return the cached response without re-execution.
Idempotency Key Contract
Header: X-Idempotency-Key: <UUIDv4>
 
Rules:
  1. Key must be UUIDv4 format (RFC 4122). Others rejected with 400.
  2. Key is scoped to (user_id, key). Cross-user reuse is blocked.
  3. Cache TTL: 24 hours from first request.
  4. Status PROCESSING: concurrent duplicate → 409 Conflict.
  5. Status COMPLETE:   subsequent duplicate → 200 with cached body.
  6. WhatsApp retries auto-generate key from message_id hash:
     key = UUIDv5(NAMESPACE_DNS, "brevio:" + whatsapp_message_id)

Idempotency State Machine
States: ABSENT → PROCESSING → COMPLETE | FAILED
 
ABSENT:    first request → write PROCESSING record → execute → write COMPLETE/FAILED
PROCESSING: duplicate concurrent → return 409 { error: "request_in_progress" }
COMPLETE:  duplicate → return 200 with cached_response_body
FAILED:    duplicate → return cached error response (preserves original HTTP status)


2.8.1 Architecture
When an agent turn exhausts the context window or exceeds a token threshold, the compaction engine triggers: in-memory message buffers and tool output summaries are compressed into a condensed context snapshot, existing stream buffers are flushed and reset, and the LLM call is retried with the compacted context. This prevents duplicate output on retry.
Compaction Trigger Conditions

Compaction Algorithm
function compact(session, buffer):
  // 1. Emit compaction stream event (for UI: "Summarizing context...")
  emit(COMPACTION_STARTED, session_id)
 
  // 2. Summarize tool outputs
  tool_summary = summarizeToolHistory(buffer.tool_results, max_tokens=2000)
 
  // 3. Summarize conversation history
  conversation_summary = summarizeMessages(buffer.messages, max_tokens=3000)
 
  // 4. Reset in-memory buffers (prevents duplicate output on retry)
  buffer.messages = []
  buffer.tool_results = []
  buffer.stream_chunks = []
 
  // 5. Rebuild context from summary
  buffer.messages = [
    { role: "system", content: session.system_prompt },
    { role: "assistant", content: f"[COMPACTED CONTEXT] {conversation_summary}" },
    { role: "assistant", content: f"[TOOL HISTORY] {tool_summary}" },
  ]
 
  emit(COMPACTION_COMPLETE, session_id)
  return buffer


2.9.1 Architecture
The system maintains a pool of model authentication profiles per provider (OAuth profiles + API keys). Profiles are selected using a weighted round-robin with rate-limit awareness. When a profile exhausts its rate limit or errors, it is deprioritized with exponential backoff and the next available profile is promoted.
Profile Selection Algorithm
function selectProfile(provider, model):
  profiles = getActiveProfiles(provider)
    |> filter(p => p.status == ACTIVE)
    |> filter(p => p.rate_limit_resets_at < now() OR p.current_rpm < p.max_rpm)
    |> sortBy(p => p.weight * (1 - p.current_utilization))
 
  if profiles.empty:
    // All profiles exhausted — wait for soonest reset
    soonest = min(allProfiles.rate_limit_resets_at)
    raise RateLimitExhaustedException(retry_after=soonest)
 
  selected = profiles[0]
  selected.current_rpm++
  recordUsage(selected.id)
  return selected
 
function onProfileError(profile_id, error):
  if error.type == RATE_LIMIT:
    profile.status = RATE_LIMITED
    profile.rate_limit_resets_at = parseRetryAfter(error.headers)
  if error.type == AUTH_EXPIRED:
    profile.status = AUTH_FAILED
    alertOps(profile_id, "OAuth token expired — refresh required")
  if error.type == QUOTA_EXCEEDED:
    profile.status = QUOTA_EXHAUSTED
    alertOps(profile_id, "Monthly quota exhausted")

2.11.1 Architecture
NO_REPLY is a reserved response token that an agent emits to signal a silent turn — no outbound message should be delivered to the user. The Brain plane filters this from all outbound payloads automatically. This is required for cron job turns, background async tool chains, and A2A coordination turns.
NO_REPLY Token Spec
Token value (exact string, case-sensitive):
  "##NO_REPLY##"
 
Detection logic:
  response_text.trim() == "##NO_REPLY##"
  OR
  response_text.trim().startsWith("##NO_REPLY##")
    // allows: "##NO_REPLY## [optional debug suffix not sent to user]"
 
On detection:
  1. Set message.outbound_suppressed = true
  2. Log no_reply_event(session_id, turn_id, trigger_reason)
  3. Execute hook: message_sending with suppression=true → hook sees suppression flag
  4. DO NOT call delivery service
  5. Continue session lifecycle normally (session remains open)

SECTION 5 — WORKFLOW SPECIFICATIONS (Temporal)
All workflows run on Temporal 1.24. Task queue: brevio-main. All retry policies use deterministic jitter formula: delay_ms = base_ms * 2^attempt + random_int(0, floor(base_ms * 0.1)). Workflow IDs follow pattern: {WorkflowType}-{entity_id}-{timestamp_ms}.

5.1 HookDispatchWorkflow
WorkflowID:     HookDispatch-{session_id}-{turn_id}
TaskQueue:      brevio-hooks
ExecutionTimeout: 60s
 
Activities:
  LoadHookRegistrations(hook_point) → []HookRegistration
    retry: max_attempts=3, initial_interval=500ms, backoff=2.0, jitter=50ms
  ExecuteHookSync(registration, context) → HookResult
    retry: max_attempts=1 (hooks are not retried — fail fast)
    timeout: registration.timeout_ms
  ExecuteHooksAsync(registrations, context) → []HookResult
    retry: max_attempts=1
    timeout: 30s (barrier timeout)
  PersistHookExecution(hook_registration_id, result) → void
    retry: max_attempts=5, initial_interval=200ms, backoff=2.0, jitter=20ms
 
State machine: see Section 2.1.1 hook dispatch algorithm
Replay safety: all hook payloads are deterministic snapshots; no live DB reads in workflow code

5.2 DMPairingWorkflow
WorkflowID:     DMPairing-{tenant_id}-{sender_id}-{platform}
TaskQueue:      brevio-gateway
ExecutionTimeout: 16min (code TTL 15min + 1min drain)
 
Activities:
  CreatePairingRequest(tenant_id, sender_id, platform) → PairingRequest
    retry: max_attempts=3, initial_interval=300ms
  SendPairingCodeMessage(sender_id, platform, code) → DeliveryReceipt
    retry: max_attempts=5, initial_interval=1s, backoff=2.0
  WaitForApproval() → ApprovalResult  [signal-driven, timeout=15min]
  AddToAllowlist(tenant_id, sender_id, platform, approved_by) → AllowlistEntry
    retry: max_attempts=3, initial_interval=300ms
  UpdatePairingStatus(id, status) → void
    retry: max_attempts=3, initial_interval=300ms
  WritePairingAudit(event_type, metadata) → void
    retry: max_attempts=5, initial_interval=200ms
 
Signals:
  approve_pairing(operator_id, submitted_code) → void
  reject_pairing(operator_id, reason) → void
 
Queries:
  get_status() → PairingStatus
 
Compensation:
  On EXPIRED: UpdatePairingStatus(EXPIRED) + WritePairingAudit(EXPIRED)
  On REJECTED (3 bad attempts): UpdatePairingStatus(REJECTED) + WritePairingAudit(REJECTED)
    + schedule_reopen = now + 24h (child workflow)

5.3 A2ACoordinationWorkflow
WorkflowID:     A2ACoord-{requester_session_id}-{idempotency_key}
TaskQueue:      brevio-brain
ExecutionTimeout: 10min (max loop depth 10 * 30s turn timeout + margin)
 
Activities:
  ResolveTargetSession(target_session_id) → AgentSession
    retry: max_attempts=3, initial_interval=300ms
  SendA2AMessage(target_session_id, payload, idempotency_key) → A2AMessage
    retry: max_attempts=3, initial_interval=500ms
  WaitForReply(target_session_id, timeout=30s) → A2AReply
    retry: max_attempts=1 (timeout = REPLY_SKIP)
  ProcessReply(reply) → ProcessedReply
    retry: max_attempts=2, initial_interval=200ms
  EmitAnnouncement(session_id, result, announced_to) → void
    retry: max_attempts=3, initial_interval=500ms
 
Invariants:
  - loop_depth incremented on each round trip; halt at 10
  - WaitForReply timeout → automatic REPLY_SKIP
  - Same message hash detection → immediate REPLY_SKIP

5.4 QueueLaneWorkflow
WorkflowID:     QueueLane-{session_id}

WorkflowID:     DoctorRun-{run_id}
TaskQueue:      brevio-cli

Activities:

SECTION 6 — POLICY DEFINITIONS (OPA)
All OPA policies live in rego/. Policy bundle is compiled and signed on each CI build. The Temporal worker and Brain service query OPA via sidecar on localhost:8181.

6.1 Hook Execution Policy
package brevio.hooks
 
import future.keywords.if
import future.keywords.in
 
# Hook execution is allowed if:
#  1. Plugin is enabled
#  2. Hook registration is enabled
#  3. Hook point is a valid enum value
#  4. Session tenant matches hook registration tenant (or hook is global)
 
default allow_hook_execution = false
 
allow_hook_execution if {
  input.plugin.is_enabled == true
  input.registration.is_enabled == true
  input.hook_point in valid_hook_points
}
 
valid_hook_points := {
  "before_model_resolve", "before_prompt_build", "before_tool_call",
  "after_tool_call", "tool_result_persist", "message_received",
  "message_sending", "message_sent", "session_start", "session_end",
  "gateway_start", "gateway_stop", "agent_bootstrap"
}
 
# HALT is only allowed if registration.can_halt = true
allow_halt if {
  allow_hook_execution
  input.registration.can_halt == true
}

6.2 DM Access Policy
package brevio.dm
 
import future.keywords.if
 
default allow_dm_message = false
 
# Allow if sender is on allowlist
allow_dm_message if {
  input.dm_policy.mode == "pairing"
  some entry in input.allowlist
  entry.sender_id == input.sender_id
  entry.platform  == input.platform
  entry.is_active == true
}
 
# Allow if policy is open (explicit opt-in)
allow_dm_message if {
  input.dm_policy.mode == "open"
  input.dm_policy.explicit_open_ack == true
}
 
# Deny with reason
deny_reason = "sender_not_paired" if {
  not allow_dm_message
  input.dm_policy.mode == "pairing"
}
 
deny_reason = "open_mode_missing_ack" if {
  not allow_dm_message
  input.dm_policy.mode == "open"
  input.dm_policy.explicit_open_ack != true
}

6.3 Tool Execution Policy (Sandbox)
package brevio.sandbox
 
import future.keywords.if
import future.keywords.in
 
default allow_tool = false
 
# Direct sessions: all tools allowed
allow_tool if {
  input.session.session_type == "direct"
}
 
# Sandboxed sessions: tool must be in allowed_tools and not in denied_tools
allow_tool if {
  input.session.session_type != "direct"
  input.tool_name in input.sandbox_config.allowed_tools
  not input.tool_name in input.sandbox_config.denied_tools
}
 
# Runtime overrides (session_tool_policies) take precedence
allow_tool if {
  some policy in input.runtime_overrides
  policy.tool_name == input.tool_name
  policy.policy == "allow"
}
 
deny_tool if {
  some policy in input.runtime_overrides
  policy.tool_name == input.tool_name
  policy.policy == "deny"
}

6.4 Model Profile Selection Policy
package brevio.model_routing
 
import future.keywords.if
import future.keywords.in
 
# A profile is eligible if:
#   status == ACTIVE
#   current_rpm < max_rpm OR rate_limit_resets_at < now
 
eligible_profiles[profile] if {
  some profile in input.profiles
  profile.status == "ACTIVE"
  profile.current_rpm < profile.max_rpm
}
 
eligible_profiles[profile] if {
  some profile in input.profiles

7.4 NO_REPLY Token — Output Verification
function verifyNoReply(response_text):
  trimmed = response_text.trim()
 
  // Primary match
  if trimmed == "##NO_REPLY##": return { is_no_reply: true, debug_suffix: null }
 
  // Prefixed match (allows debug suffix)
  if trimmed.startsWith("##NO_REPLY##"):
    debug = trimmed.slice(len("##NO_REPLY##")).trim()
    return { is_no_reply: true, debug_suffix: debug }
 
  return { is_no_reply: false }
 
// Note: case-sensitive. "##no_reply##" is NOT matched.
// Guardrail: if response contains ##NO_REPLY## anywhere except at start → log WARNING,
//            do NOT suppress (may be intentional user text discussing the token).

9.2 Migration Safety Rules
Forbidden in non-backward-compatible migrations:
  - DROP TABLE
  - DROP COLUMN
  - ALTER COLUMN type change (incompatible cast)
  - RENAME TABLE or RENAME COLUMN
  - Adding NOT NULL without DEFAULT on existing table
 
Allowed without code deploy:
  - CREATE TABLE (new table)
  - CREATE INDEX CONCURRENTLY
  - Adding NULLABLE column
  - Adding DEFAULT to existing column
 
Two-phase deployment for breaking changes:
  Phase 1: Add new column/table (backward compatible)
  Phase 2: Remove old column after all consumers migrated

9.3 Rollback Strategy
Trigger conditions for automatic rollback:
  - Health check POST /health/doctor/run returns CRITICAL within 5min of deploy
  - Error rate > 1% sustained for 2min (Datadog SLO breach)
  - p99 latency > 5s sustained for 2min
 
Rollback procedure (automated):
  1. Helm rollback brevio-{service} --wait
  2. If DB migration was applied: run DOWN migration (reversible only)
  3. Alert PagerDuty: "Auto-rollback executed for {service} at {timestamp}"
  4. Hold next deploy until operator acks in Slack: /brevio deploy-unblock

A failure domain is a set of components that can fail together without causing failure in components outside the domain. The following defines all Brevio failure domains, their blast radius, and isolation mechanisms.


Cross-Domain Failure Invariants
INVARIANT FD-INV-001: FD-05 (Redis) failure must NOT cause FD-01 (Gateway) crash.
  Enforcement: Gateway uses Redis with connection timeout=2s, fallback=degraded mode.
  Degraded mode: idempotency checks skipped (logged), hook bus disabled (hooks skip),
  queue lanes managed in-memory (volatile, lost on restart).
 
INVARIANT FD-INV-002: FD-04 (PostgreSQL) failure must NOT cause FD-06 (Temporal) crash.
  Enforcement: Temporal activities have max_attempts; DB retry bounded at 5x.
  After 5x failure: activity fails → workflow DLQ. Temporal itself uses its own DB.
 
INVARIANT FD-INV-003: FD-08 (sandbox pool) failure must NOT affect FD-01 (direct sessions).
  Enforcement: Sandbox containers are isolated from gateway pod networking.
  Direct sessions never contact sandbox pool.
 
INVARIANT FD-INV-004: No single FD failure may cascade to data loss.
  Enforcement: All writes go to PostgreSQL (durable). Redis is cache-only.
  Temporal event history is source of truth for workflow state.

Versioning Strategy
URL versioning: /v{major} (e.g. /v1/hooks/plugins)
Current version: v1
 
Version lifecycle:
  CURRENT → DEPRECATED (6-month notice) → SUNSET (removed)
 
Deprecation procedure:
  1. Add Deprecation: date="YYYY-MM-DD" header to all deprecated endpoint responses
  2. Add Sunset: date="YYYY-MM-DD" header
  3. Log deprecation_warning per request in observability
  4. Notify all active API consumers via email + Slack 6 months before sunset
  5. On sunset date: return 410 Gone for deprecated version endpoints

10.1 A2A Message Processing Prompt
SYSTEM:
You are a sub-agent in the Brevio AI assistant platform responding to an agent-to-agent
(A2A) coordination request. Process the incoming request, execute any required actions,
and return a structured response. If no further coordination is needed, end your response
with the exact token: REPLY_SKIP
 
USER:
A2A_REQUEST from session {requester_session_id}:
{a2a_payload_json}
 
Your context:
{session_context_summary}
 
Respond with your result. If this completes the coordination, append REPLY_SKIP on the final line.

A2A Processing — Input Schema
{
  "requester_session_id": "uuid",
  "a2a_payload_json": "string (JSON object: {task, context, expected_output_format})",
  "session_context_summary": "string (max 2000 tokens)"
}

A2A Processing — Output Verification Contract
Validations:
  1. Response must not be empty (min 10 chars)
  2. If ending in REPLY_SKIP: trim last line must == "REPLY_SKIP"
  3. Must not contain: raw internal identifiers (UUIDs in system format)
     (signals agent leaked internal state to user-facing output)
  4. Max length: 8000 chars. If exceeded: truncate to 8000 + " [TRUNCATED]"
 
On validation failure:
  - Return error to requester session: { error: "A2A_RESPONSE_INVALID", detail: reason }
  - Do NOT retry A2A processing (idempotency key prevents dedup issues)

10.2 Bootstrap Context Assembly Prompt
SYSTEM:
You are the context assembly engine for the Brevio AI assistant. Your task is to
synthesize a concise user context summary from the provided profile and recent history.
This summary will be injected into the agent system prompt. Be factual, structured,
and do not add commentary. Output ONLY the summary JSON object.
 
USER:
User profile: {user_profile_json}
Recent session summaries (last 3): {recent_summaries_json}
Temporal state (pending tasks, goals): {temporal_state_json}
 
Produce a JSON object with this exact schema:
{
  "user_summary": "string (2-3 sentences, factual user context)",
  "active_goals": ["string", ...],   // max 5
  "pending_tasks": ["string", ...],  // max 5
  "context_notes": "string"         // anything critical the agent must know
}

Bootstrap Context — Input Schema
{
  "user_profile_json": "string (JSON: name, preferences, subscription_tier, timezone)",
  "recent_summaries_json": "string (JSON array of past session summary strings)",
  "temporal_state_json": "string (JSON: active goals, pending tasks from memory tables)"
}

Bootstrap Context — Output Verification Contract
Validations:
  1. Output must be valid JSON (JSON.parse() succeeds)
  2. Must contain all 4 required keys: user_summary, active_goals, pending_tasks, context_notes
  3. user_summary: string, 10-500 chars
  4. active_goals: array, max 5 items, each item string < 200 chars
  5. pending_tasks: array, max 5 items, each item string < 200 chars
  6. context_notes: string, max 1000 chars
  7. Must not contain: "I cannot", "I don't have", "As an AI" (model error indicators)
 
On validation failure:
  Retry once with simplified prompt (omit temporal_state_json)
  On second failure: use fallback = { user_summary: "Context unavailable.", active_goals: [], pending_tasks: [], context_notes: "" }

10.3 Tool Routing Classification Prompt
SYSTEM:
You are a tool routing classifier for the Brevio AI assistant platform.
Given a user message, classify which tools from the available tool set are required.
Return ONLY a JSON object. No preamble. No commentary.
 
USER:
Available tools: {available_tools_json}
User message: {user_message}
Session context: {session_context_brief}
 
Return JSON: { "required_tools": ["tool_name", ...], "confidence": 0.0-1.0,
               "reasoning": "one sentence" }

Tool Routing — Output Verification Contract
Validations:
  1. Valid JSON
  2. required_tools: array of strings, each must be in available_tools_json
     (unknown tool names rejected — prevents hallucinated tool calls)
  3. confidence: number 0.0-1.0
  4. reasoning: string, max 200 chars
  5. If confidence < 0.4: treat as no_tools_required (log low_confidence routing)
 
On validation failure:
  Fallback: required_tools = [] (no tools), confidence = 0.0
  This is safe: agent proceeds without pre-routed tools, may call tools during turn

Supply Chain Integrity Enforcement (complete)
# Build phase — container images
1. All container images built via GitHub Actions only (branch protection enforced)
2. Multi-stage builds: builder stage (full toolchain) → runtime stage (minimal base)
   Base images: distroless/base-nossl:latest (pinned by digest, not tag)
3. cosign sign: each image signed at build time with Sigstore keyless signing
   cosign sign --key awskms://alias/brevio-signing-key ghcr.io/brevio/{service}:{tag}
4. Syft SBOM generated: syft ghcr.io/brevio/{service}:{tag} -o spdx-json
5. SBOM attached as OCI attestation: cosign attest --type spdxjson
 
# Verification phase — deployment
6. Kyverno policy: verify-image-signatures
   verifyImages:
     - imageReferences: ["ghcr.io/brevio/*"]
       attestors:
         - entries:
             - keyless:
                 subject: "https://github.com/brevio/*/.github/workflows/*.yml@refs/heads/main"
                 issuer: "https://token.actions.githubusercontent.com"
 
7. Kyverno policy: require-sbom-attestation
   verifyImages:
     - imageReferences: ["ghcr.io/brevio/*"]
       attestations:
         - predicateType: "https://spdx.dev/Document"
           conditions:
             - all: [{ key: "{{spdxVersion}}", operator: NotEquals, value: "" }]
 
# Dependency integrity
8. npm: package-lock.json committed; npm ci used in CI (not npm install)
9. pip: requirements.txt pinned to exact versions with hashes:
   pip install --require-hashes -r requirements.txt
10. Go: go.sum committed; go mod verify in CI
 
# Artifact signing chain summary
Developer commit → GitHub Actions (OIDC) → cosign keyless sign (Fulcio CA)
→ Rekor transparency log entry → OCI registry with signature + SBOM attestation
→ Kyverno admission controller verifies signature + SBOM before pod scheduled
→ No unsigned image can run in production cluster (admission webhook enforces)

Restore Procedures
PostgreSQL Point-in-Time Recovery (PITR)
# Restore PostgreSQL to specific point in time
1. Identify restore target time: incident_occurred_at - 5min (before corruption)
2. In AWS Console (or CLI):
   aws rds restore-db-instance-to-point-in-time \
     --source-db-instance-identifier brevio-prod-db \
     --target-db-instance-identifier brevio-prod-db-restored \
     --restore-time 2026-01-01T12:00:00Z
3. Update Kubernetes secret: brevio-db-host → restored instance endpoint
4. Restart all pods: kubectl rollout restart deployment -n brevio-production
5. Verify: brevio-cli doctor (HC-005 should pass)
6. Decommission original instance after 48h validation window
 
RTO for PITR: < 30min
RPO for PITR: < 5min (WAL archiving continuous)

Vault Snapshot Restore
# Restore Vault from snapshot
1. Stop Vault writes: vault operator seal (if compromise detected)
2. Get latest snapshot:
   aws s3 ls s3://brevio-vault-backups/ --recursive | sort | tail -1
3. Restore:
   vault operator raft snapshot restore latest-snapshot.bak
4. Unseal: vault operator unseal (3 of 5 key shares required)
5. Verify: vault status + brevio-cli doctor (HC-006)
6. Notify all services to re-auth (Vault AppRole tokens invalidated)
 
RTO for Vault restore: < 10min
RPO for Vault restore: < 6h (snapshot frequency)

Redis Restore
# Redis data is cache-only for hooks bus and idempotency.
# On full Redis loss:
1. Provision new Redis cluster from ElastiCache (< 5min)
2. Update K8s ConfigMaps with new endpoint
3. Restart affected pods (gateway, brain, worker)
4. Accept: idempotency keys for last 24h are lost (safe — WhatsApp retries get new processing)
5. Accept: in-flight queue lane messages lost (safe — WhatsApp retries via idempotency)
6. Verify: brevio-cli doctor (HC-010)
 
RTO for Redis: < 10min
RPO for Redis: Last 6h RDB snapshot (for warm reload); cold start acceptable


IR-01: Vault Sealed / Unavailable
SYMPTOMS: All services return 503. brevio-cli doctor HC-006 FAIL.
          Logs: "vault: connection refused" or "vault: permission denied".
 
IMMEDIATE STEPS:
1. Check Vault cluster status: vault status (from ops bastion)
2. If sealed (initialized=true, sealed=true):
   vault operator unseal  (requires 3 of 5 key shares)
   Contact on-call key holders (PagerDuty: "Vault unseal required")
3. If unavailable (node crash): check Vault HA — other nodes should auto-elect leader
   vault operator raft list-peers  (verify quorum)
4. If all nodes down: restore from snapshot (see Backup Runbook)
5. After recovery: brevio-cli doctor  (confirm HC-006 PASS)
6. Monitor for 10min: no new 503s from services
RTO TARGET: < 15min

IR-02: Hook Bus (Redis) Unavailable
SYMPTOMS: Hook dispatch failing. brevio-cli doctor HC-004 FAIL.
          Logs: "redis: connection timeout" from brain service.
 
IMMEDIATE STEPS:
1. Check Redis cluster: redis-cli -h brevio-cache PING (from pod)
2. If single shard down: ElastiCache auto-failover should promote replica (< 2min)
3. If all shards down: services enter degraded mode automatically (see FD-INV-001)
   - Hooks skip (non-blocking)
   - Queue lanes managed in-memory
   - Idempotency checks skipped (logged)
4. Provision replacement Redis if needed: terraform apply (brevio-cache module)
5. Update K8s ConfigMaps + restart pods
6. brevio-cli doctor HC-004 + HC-010 must PASS before declaring resolved
RTO TARGET: < 10min (auto-failover 2min, manual 10min)

IR-03: Sandbox Container Escape Detected
SYMPTOMS: CloudTrail anomaly alert OR container audit log shows unauthorized host access.
          Container process reading files outside /workspace or /tools.
 
IMMEDIATE STEPS (SECURITY INCIDENT — P1):
1. IMMEDIATELY: kubectl cordon all sandbox nodes (prevent new scheduling)
   kubectl cordon -l brevio/sandbox-node=true
2. Kill all running sandbox containers:
   kubectl delete pods -n brevio -l brevio/sandbox=true
3. Disable all group/channel sessions: brevio-cli sessions disable-sandboxed
4. Isolate affected nodes: revoke network access at security group level (sg-sandbox)
5. Preserve evidence: docker logs <container_id> before killing
6. Forensic analysis: identify which session triggered escape, what was accessed
7. Patch and re-image sandbox nodes before re-enabling
8. File security incident report. Legal review required if PII was accessed.
RTO TARGET: 30min to full isolation. Re-enable only after security sign-off.

IR-04: Mass Pairing Code Flood (DDoS via Unknown Senders)
SYMPTOMS: dm_pairing_requests table growing rapidly. Gateway CPU elevated.
          Logs: high volume of CODE_ISSUED audit events from varied sender_ids.
 
IMMEDIATE STEPS:
1. Enable rate limiting at ALB level: aws wafv2 update-rule-group (add sender_id rate limit)
2. Check pairing_requests volume: SELECT COUNT(*) FROM dm_pairing_requests
   WHERE status='PENDING' AND created_at > now() - interval '1 hour'
3. If > 1000 PENDING: trigger bulk expiry: UPDATE dm_pairing_requests
   SET status='EXPIRED' WHERE status='PENDING' AND created_at < now() - interval '5 minutes'
4. Add IP-based block at ALB WAF if attack source identifiable
5. Consider switching dm_policy to open + IP allowlist temporarily (operator decision)
6. Monitor: pairing rate should drop to < 10/min after mitigation
RTO TARGET: < 15min to traffic reduction

IR-05: Skill Gate Blocking All Sessions (mis-deployed config)
SYMPTOMS: All sessions start but no skills load. Agents have minimal context.
          brevio-cli doctor HC-011 FAIL. Logs: all skill gate evals = FAIL.
 
IMMEDIATE STEPS:
1. Identify cause: check skill frontmatter for recently changed skills
   brevio-cli skills list --gate-status  (shows which prerequisites failing)
2. If env var missing: verify K8s secrets/configmaps for affected env vars
3. If binary missing: check Pod image (may need rebuild with missing binary)
4. Emergency override: set metadata.openclaw.always: true on critical skills
   (deployed via brevio-cli skills patch --id {id} --always-load true)
5. Force skill reload: brevio-cli sessions reload-skills --all-active
6. Monitor: skill_gate_evaluations.result = PASS rate should recover
RTO TARGET: < 20min

IR-06: A2A Deadlock — All Sessions Stuck at Max Loop Depth
SYMPTOMS: a2a_loop_depth p99 = 10 sustained 5min. Sessions not completing.
          A2ACoordinationWorkflow all in WAITING_REPLY state.
 
IMMEDIATE STEPS:
1. Identify stuck workflows: temporal workflow list --query "WorkflowType='A2ACoord' AND ExecutionStatus='Running'"
2. Terminate stuck workflows (they will auto-emit REPLY_SKIP on cancel):
   temporal workflow terminate --query "WorkflowType='A2ACoord'" --reason "deadlock recovery"
3. Sessions should recover: QueueLane signals run_complete, resumes normal processing
4. Identify root cause: check loop_detection code (message hash check in Brain)
5. If code bug: hotfix and deploy brevio-brain
6. Verify: a2a_loop_depth p99 drops; sessions complete normally
RTO TARGET: < 20min

IR-07: Compaction In-Progress Stuck (INV-007 violation)
SYMPTOMS: compaction_state rows with status=IN_PROGRESS older than 5min.
          Sessions unable to proceed (compaction blocking turn execution).
 
IMMEDIATE STEPS:
1. Background cleanup job should auto-resolve (runs every 30s): wait 2 min.
2. If not resolved: manually mark stuck compactions as FAILED:
   UPDATE compaction_state SET status='FAILED'
   WHERE status='IN_PROGRESS' AND started_at < now() - interval '5 minutes'
3. Sessions will use fallback: truncated context (safe degraded mode)
4. Identify root cause: check SummarizeMessages LLM call — likely timeout or rate limit
5. If LLM rate limited: check model profile status (run doctor HC-007)
RTO TARGET: < 5min (auto-resolved by cleanup job in most cases)
