# v0.6.0 Architecture ŌĆö `ContextSource` Seam

**Status:** founder-approved 2026-06-08 (before v0.6.0C coding begins). This doc captures the locked design so the seam is preserved across the Calendar phase and any future read-only context source.

**Tier:** TIER 3 docs-only. No runtime code. No registry. No generic framework. **Build the seam, defer the system.**

**Principle:** the agent core reasons over **normalized context packets**; provider-specific logic lives at the adapter/source boundary. Adding a new source must not require changes to the agent core.

---

## 1. Long-term separation (already locked by founder)

| Side | Examples | Lifecycle |
|---|---|---|
| **Read-only context sources** | email, calendar, contacts, web, files, messages | fetch ŌåÆ ContextPacket (privacy-projected) ŌåÆ ranker / HMR / reasoning |
| **Action tools** (FUTURE) | send text, send email, create calendar event, book, pay, submit form, browser automation | `ActionProposal` ŌåÆ Policy Gate ŌåÆ User Approval ŌåÆ Execute ŌåÆ Audit |

v0.6.0 ships **read-only** only. Action tools / Tool Gateway / browser automation are explicitly **future-only** (see ┬¦7ŌĆō┬¦8).

## 2. `ContextSource<T>` ŌĆö the seam (interface only)

```ts
// apps/fomo/src/core/context-source.ts (NEW at v0.6.0C)

/**
 * A normalized read-only context source. One implementation per
 * provider/domain (Calendar, future Contacts, future Web, etc.).
 *
 * The seam is intentionally minimal ŌĆö no registry, no dispatch table,
 * no factory. Each source is hand-wired by the consumer (today: the
 * polling worker) until the codebase has 3+ wired sources, at which
 * point a small registry may emerge from the call sites themselves.
 */
export interface ContextSource<T extends ContextPacket> {
  /** Stable identifier ('calendar', 'pil', future 'contacts', etc.) */
  readonly source_id: string;

  /**
   * Best-effort fetch. MUST return null (never throw) when:
   *   - kill switch off,
   *   - user not in allowlist,
   *   - no data to contribute,
   *   - underlying adapter call failed.
   * Caller treats null as "no context" and proceeds without the source.
   */
  fetch(args: ContextSourceFetchArgs): Promise<T | null>;
}

export interface ContextSourceFetchArgs {
  readonly user_id: string;
  // Hints ŌĆö sources ignore fields they do not consume. The caller does
  // NOT need to know which fields a given source uses.
  readonly sender_email_hash?: string | null;
  readonly window_hours?: number | null;
  readonly now?: () => Date;
}
```

## 3. `ContextPacket` ŌĆö the normalized shape

```ts
export interface ContextPacket {
  readonly source_id: string;             // matches ContextSource.source_id
  readonly provenance: ContextProvenance; // structural metadata only ŌĆö safe for audit detail
  readonly projection: Readonly<Record<string, unknown>>; // source-specific payload
}

export interface ContextProvenance {
  readonly source_id: string;
  readonly fetched_at_iso: string;
  readonly kill_switch_on_at_fetch: boolean;
  readonly window_hours?: number;
  /**
   * Bounded structural count (e.g. event_count_in_window). NEVER the
   * raw content count of sensitive items. Always safe to surface in
   * audit detail.
   */
  readonly item_count?: number;
}
```

**Privacy contract on `projection`:** the adapter discards the raw provider response before producing the packet. Whatever lands in `projection` IS the field-exclusion allowlist for that source. No "raw event body", no "full message text", ever.

## 4. `CalendarContextPacket` ŌĆö first concrete implementation

```ts
export interface CalendarContextPacket extends ContextPacket {
  readonly source_id: 'calendar';
  readonly projection: {
    readonly events: readonly CalendarEventProjection[];
    readonly window_hours: number;
    readonly nearest_event_start_offset_minutes: number | null;
  };
}

export interface CalendarEventProjection {
  readonly summary: string;     // 'Busy' deterministic when visibility=private
  readonly start_iso: string;
  readonly end_iso: string;
}
```

`summary` lives in the packet because v0.6.0E HMR needs it to render "before your board meeting tomorrow morning." It is **NEVER** in the `brevio.rank.calendar_applied` audit detail ŌĆö that carries only the bounded structural fields from `provenance` + `nearest_event_start_offset_minutes`.

## 5. Adapter privacy projection (field exclusion at the boundary)

```
GoogleCalendarClient.listEvents()
  ŌåÆ raw Google API response (contains attendees, description, location, conferenceData, etc.)
  ŌåÆ CalendarContextSource.projectToPacket()   ŌåÉ THE BOUNDARY
       DISCARDS: attendees, description, location, conferenceData,
                 attachments, recurrence, visibility (used to decide
                 'Busy' replacement, then dropped), colorId, etag,
                 htmlLink, organizer, creator, every other field
       PROJECTS: summary (or 'Busy'), start_iso, end_iso
  ŌåÆ CalendarContextPacket (frozen)
  ŌåÆ returned to caller; raw response is unreachable
```

**Test requirement at v0.6.0C:** given a raw Google response with `attendees` / `description` / `location` / `conferenceData` set, the produced `CalendarContextPacket` has none of those fields **and** no substring of those raw values appears in any field of the packet (privacy canary).

## 6. Where Calendar context enters (no coupling to FOMO ranker internals)

```ts
// v0.6.0D shadow + v0.6.0E live
export interface RankEmailRequest {
  readonly raw: RawEmailContext;
  readonly user_id: string;
  readonly pil_context: PilContext | null;

  /**
   * v0.6.0 ŌĆö normalized array of context packets from any source.
   * The ranker's prompt builder consumes packet shapes it has been
   * written for; unknown source_ids are silently ignored (additive,
   * never blocking).
   */
  readonly context_packets?: readonly ContextPacket[];
}
```

**The agent core / ranker code:**
- does NOT import the Google Calendar adapter
- does NOT call `calendar.events.list`
- does NOT know Google API specifics (auth, scope, pagination, etc.)
- reads `context_packets`, type-narrows by `source_id`, calls the per-source prompt-block converter

**The polling worker (per rank, when enabled + allowlisted):**
1. Calls each wired `ContextSource.fetch()` (Calendar at v0.6.0C; PIL conceptually maps here but stays in its current shape ŌĆö see ┬¦7)
2. Filters out `null` returns
3. Passes the array into the rank request

This is the seam. Provider-specific logic lives in the adapter + source; the agent core reasons over normalized packets.

## 7. Future source onboarding pattern

When the next source lands (Contacts, Web, Files, Messages, ŌĆ”):

1. Define `XContextPacket extends ContextPacket` with the locked projection shape.
2. Implement `XContextSource implements ContextSource<XContextPacket>` with privacy projection at the adapter boundary.
3. Wire the new source into the worker's per-rank fetch list (one line in `gmail-poll.ts`).
4. Add a per-packet prompt-block converter in the ranker (one new function).

**No changes to the agent core. No changes to `ContextSource` interface. No changes to PIL. No changes to Calendar.**

When there are Ōēź3 wired sources, the worker's hand-wired list can graduate into a small registry. **Not a v0.6.0 problem.**

## 8. Intentionally deferred (NOT in v0.6.0)

- **No `ContextSource` registry / dispatch table.** Worker hand-wires Calendar.
- **No PIL refactor onto `ContextSource`.** PIL stays as `buildLivePilContext`. The conceptual mapping is documented; no code change.
- **No `ActionProposal` type.** Tool-side; deferred.
- **No Tool Gateway.** See ┬¦9.
- **No `ContextProvenance` persistence.** Packets are ephemeral; provenance lands only in the per-rank audit detail (`brevio.rank.calendar_applied`).
- **No multi-source merge logic in the agent core.** Each source emits its own packet; the ranker prompt builder handles them independently.
- **No `windowHours` per-intent dispatch logic.** v0.6.0C accepts `windowHours` per call; the worker passes the 48h default. Intent-based windows are forward-noted; the seam already supports them via the parameter.
- **No multi-calendar, attendees, descriptions, location** (locked per v0.6.0B).

## 9. What later becomes the Tool Gateway (FUTURE-only)

Symmetric to `ContextSource` but on the action side:

| | READ-ONLY (v0.6.0+) | WRITE / ACTION (future) |
|---|---|---|
| **Interface** | `ContextSource<T>` | `ActionTool<TProposal, TResult>` |
| **Output** | `ContextPacket` | `ActionProposal` ŌåÆ `ActionResult` |
| **Lifecycle** | `fetch()` best-effort | propose ŌåÆ policy_gate ŌåÆ **user_approval** ŌåÆ execute ŌåÆ audit |
| **Privacy** | field-exclusion projection at boundary | risk classification + reversibility metadata on proposal |
| **Failure** | `null` (no context) | `ActionFailed { reason, recoverable }` |

**v0.6.0 ships the left column only.** The right column is a future architecture requirement documented for seam alignment ŌĆö NOT for code.

When the first new write/action tool lands (likely a calendar event creator, an email composer, or a simple browser-automated form filler), THAT phase introduces `ActionProposal`, `ActionTool`, and the policy/approval/execute/audit lifecycle. **It does not retrofit existing tools** (`gmail.read`, `sendblue.send_user_message` stay on today's Permission Gate dispatch).

## 10. Browser automation ŌĆö long-term posture confirmation

Per founder direction, restated for the record so v0.6.0's seam doesn't trap future browser automation in one-off Calendar shape:

**API first ŌåÆ trusted MCP/adapter second ŌåÆ sandboxed browser automation only when necessary.**

When browser automation eventually lands (post-Tool-Gateway), it MUST be:
- sandboxed
- permission-gated
- kill-switched
- audited
- visually / state verified
- reversible where possible
- limited to approved domains / tasks
- **never** allowed to silently submit, pay, book, send, or purchase
- final-action gated by explicit user approval

For high-risk actions (payments, checkout, bookings, transfers, purchases): Brevio may **prepare** the action, but the final step requires explicit user approval via `ActionProposal ŌåÆ Policy Gate ŌåÆ User Approval ŌåÆ Execute ŌåÆ Audit`.

**Not in v0.6.0. Not in v0.7.0.**

## 12. Approved decisions inherited

From v0.6.0A gate + clarifications + v0.6.0B doc:
- Scope: `https://www.googleapis.com/auth/calendar.events.readonly`, treated as OAuth-sensitive until Cloud Console confirms classification
- Primary calendar only at v0.6.0C
- Default `windowHours = 48` from env; **configurable per call**; **non-default value tests mandatory**
- Fields IN: `summary`, `start`, `end` (only)
- Fields deferred: `location`
- Fields OUT: `attendees`, `description`, `attachments`, `conferenceData`
- Ephemeral storage only; process-local 60s cache; **no DB writes** of event content; **no new memory_signal kind**
- `visibility = private` ŌåÆ `summary = 'Busy'` deterministic
- v0.6.0E live trigger requires v0.6.0D shadow PASS + founder taste check on rendered HMR samples
