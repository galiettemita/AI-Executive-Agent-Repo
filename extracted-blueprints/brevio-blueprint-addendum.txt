1.5 Data Flow Graph
The new categories introduce three critical data flow patterns not present in the original evaluation:

Pattern 1: Browser-Mediated Action (Browser & Automation)
User Request → Brain Plane (intent classification) → Router (select skill)
  → Browser Service (launch headless session) → Target Website
  → DOM Interaction → Result Extraction → Brain Plane (validation)
  → Gateway Plane (format response) → User

Pattern 2: Multi-Agent Fan-Out (Agent Orchestration)
User Request → Brain Plane (task decomposition) → Task Delegator
  → [Agent-1: Research] [Agent-2: Draft] [Agent-3: Validate] (parallel)
  → Parallel Executor (collect results, resolve conflicts)
  → Brain Plane (synthesize final output) → User

Pattern 3: Proactive Scheduled Execution (Cron + Memory)
Cron Trigger (8:00 AM user timezone) → Briefing Engine
  → [Calendar Skill] [Weather Skill] [News Skill] [Finance Skill] (parallel)
  → Memory Service (retrieve user preferences, past patterns)
  → Brain Plane (compose personalized briefing)
  → Gateway Plane (WhatsApp/iMessage delivery) → User

1.7 Deterministic Control Plane
All new services register with the Brevio Control Plane via a deterministic service mesh:

Service Discovery: Consul-based with health check interval = 10s, deregister_critical_after = 30s
Load Balancing: Round-robin with sticky sessions for browser automation (session affinity by execution_id)
Circuit Breaking: Half-open after 30s, trip after 5 consecutive failures, reset after 3 successful probes
Rate Limiting: Per-user token bucket — 100 requests/minute for Hands, 50/minute for Brain, 10/minute for Agent Orchestration
Timeout Cascade: Gateway=60s, Brain=45s, Hands=30s, Browser=120s (extended for page loads)

3.3 Referential Integrity Enforcement
All foreign key relationships in this addendum are enforced at the database level via FK constraints with explicit ON DELETE behavior. No application-level-only referential integrity is permitted.


Invariant: Every FK reference in this addendum points to a table that either (a) exists in the core Brevio schema (users, conversations) or (b) is created in a prior or same-numbered migration in this addendum. No orphaned references exist.

3.4 Schema Validation Invariants
The following invariants MUST hold across all migrations. These are enforced via CHECK constraints in DDL and validated in CI:

INV-001: Every table has a UUID primary key column named 'id' with DEFAULT gen_random_uuid()
INV-002: Every table has a 'user_id' column with FK to users(id) ON DELETE CASCADE
INV-003: Every table has 'created_at TIMESTAMPTZ NOT NULL DEFAULT now()'
INV-004: Every status column uses CHECK constraint with explicit enum values — no free-text status
INV-005: Every table has ROW LEVEL SECURITY enabled with user_id-based policy
INV-006: Every FK reference targets a table created in a prior or same migration number
INV-007: Every index name follows pattern idx_{table}_{column(s)}
INV-008: No table references a column that does not exist in the referenced table
INV-009: All DECIMAL columns have explicit precision/scale specifications
INV-010: All VARCHAR columns have explicit max length — no unbounded VARCHAR

4. API Contracts (OpenAPI 3.1)

4.1 Versioning Strategy
All API endpoints are versioned under /v1/ prefix. Backward compatibility rules:
New fields: Added as optional with default values — never breaking
Field removal: Deprecated for 2 minor versions before removal
Enum expansion: New enum values are additive-only — never remove existing values
Error codes: New error codes may be added but existing codes never change meaning

4.8 Idempotency Guarantees
All mutating endpoints (POST, PUT, DELETE) accept an 'idempotency_key' header or body field. Contract:
Keys are scoped to user_id + endpoint — same key on different endpoints is treated as distinct
First request with key: executed normally, result cached for 24 hours
Repeat request with same key within 24 hours: returns cached result with HTTP 200 (not re-executed)
Repeat request after 24 hours: returns 409 IDEMPOTENCY_KEY_EXPIRED
Storage: Redis with TTL 86400s, key format: idem:{user_id}:{endpoint}:{key}

5.2 Retry Policy Matrices (All Workflows)
The following retry matrices define bounded retry behavior for every workflow. All values are deterministic and replay-safe:

Deterministic Jitter Formula (applied to all retries):
actual_delay = base_delay * backoff^attempt * (1 + jitter_pct * (hash(jitter_seed + attempt) % 100) / 100)
Where jitter_pct = 0.2 for all workflows. This ensures deterministic replay — the same workflow execution always computes the same retry delay.

7. LLM Layer Contracts

7.1 Prompt Templates

7.1.1 Task Complexity Classifier (Smart Router)
System: You are a task complexity classifier for an AI assistant. Classify the user's request
into exactly one complexity level. Respond with ONLY a JSON object.

Classification rules:
- SIMPLE: Factual lookups, single-step actions, status checks, greetings
  Examples: 'What time is it?', 'Play some music', 'What's the weather?'
- MEDIUM: Multi-step tasks, light reasoning, standard workflows
  Examples: 'Schedule a meeting for next week', 'Summarize my emails', 'Track this package'
- COMPLEX: Analysis, planning, multi-service coordination, long-form generation
  Examples: 'Plan a dinner party', 'Review this contract', 'Create a marketing campaign'
- CRITICAL: High-stakes decisions, financial transactions, legal matters, multi-agent needed
  Examples: 'Analyze my investment portfolio', 'Book flights for the whole team'

User: {user_message}

Respond ONLY with: {"complexity": "SIMPLE|MEDIUM|COMPLEX|CRITICAL", "reason": "<brief>"}

7.1.2 Memory Extraction Prompt
System: Extract structured facts about the user from this conversation. Return ONLY a JSON array.

Rules:
- Extract ONLY explicitly stated facts, not inferences
- Categorize each fact: preference, fact, behavior, context
- Assign confidence: 1.0 for explicit statements, 0.8 for strong implications, 0.6 for weak signals
- Do NOT extract: temporary states, one-time requests, conversation artifacts

Conversation: {conversation_text}

Respond ONLY with:
[{"type": "preference|fact|behavior|context", "category": "<topic>",
  "content": "<fact>", "confidence": 0.0-1.0}]

7.1.3 Task Decomposition Prompt (Agent Orchestration)
System: You are a task decomposition engine. Break the user's complex request into atomic
sub-tasks that can be executed in parallel by specialized agents.

Rules:
- Each sub-task must be independently executable
- Identify dependencies between sub-tasks (which must complete before others)
- Assign each sub-task a required skill from the available skill registry
- Estimate cost category: low (<$0.01), medium ($0.01-$0.05), high (>$0.05)

Available skills: {skill_registry}
User objective: {objective}
Max agents: {max_agents}
Budget: ${max_cost_usd}

Respond ONLY with:
{"subtasks": [{"description": "...", "skill": "...", "dependencies": [],
  "cost_estimate": "low|medium|high", "priority": 1-10}],
 "execution_order": [["parallel_group_1"], ["sequential_after_1"]]}

7.1.4 Daily Briefing Composition Prompt
System: Compose a personalized {briefing_type} briefing for the user.
Format for {delivery_channel} (use appropriate formatting: WhatsApp markdown, plain text, etc.)

User context from memory: {user_preferences}
Time zone: {timezone}

12. Security & Governance

12.1 Threat Model

12.2 Trust Boundaries
Four trust boundaries exist in the new services:
TB-1: User -> API Gateway: All input is untrusted. Full schema validation, rate limiting, authentication required.
TB-2: API Gateway -> Services: Authenticated service-to-service communication via mTLS. Input pre-validated.
TB-3: Services -> External APIs (LLM providers, social platforms, booking sites): Outbound only. No inbound connections. API keys rotated per policy.
TB-4: Browser Service -> Internet: Highest risk boundary. All browser sessions are sandboxed, isolated per user, and destroyed after use. No persistent state in browser.

12.4 SBOM Definition & Supply Chain Integrity Enforcement
Every container image includes a Software Bill of Materials generated by Syft and attested by Cosign. Supply chain integrity is enforced end-to-end:

Supply Chain Integrity Enforcement:
SCI-001: All base images pinned to SHA256 digest — no mutable tags (e.g., FROM node:20@sha256:abc123...)
SCI-002: All npm dependencies use exact versions in package-lock.json with integrity hashes verified
SCI-003: Container images are signed with Cosign using keyless signing (Fulcio + Rekor transparency log)
SCI-004: SBOM attestation attached to every image via Cosign in-toto predicates
SCI-005: Kubernetes admission controller (Kyverno) rejects any unsigned or unattested image
SCI-006: CI pipeline runs npm audit and Trivy scan — build fails on HIGH/CRITICAL vulnerabilities
SCI-007: SLSA Level 3 provenance generated for all build artifacts via GitHub Actions OIDC

# Generate SBOM
syft packages brevio/browser:${TAG} -o spdx-json > sbom-browser.spdx.json

# Attest SBOM to image (keyless signing via Fulcio)
cosign attest --predicate sbom-browser.spdx.json --type spdxjson brevio/browser:${TAG}

# Sign image (keyless)
cosign sign brevio/browser:${TAG}

# Verify attestation (enforced by Kyverno admission controller in production)
cosign verify-attestation --type spdxjson brevio/browser:${TAG}
cosign verify brevio/browser:${TAG}

14. Runbooks

14.1 Deployment Runbook
Verify all CI gates pass (schema validation, tests, security scan)
Tag release: git tag -s v1.x.y -m 'Release v1.x.y'
Build containers: ./scripts/build-all.sh --tag v1.x.y
Sign artifacts: cosign sign brevio/{service}:v1.x.y
Deploy to staging: ./scripts/deploy.sh staging v1.x.y
Run smoke tests: npm run test:smoke -- --env staging
Verify SLO metrics for 30 minutes
Deploy to production (blue/green): ./scripts/deploy.sh production v1.x.y --strategy blue-green
Canary at 5% for 15 minutes: ./scripts/canary.sh production --pct 5 --duration 900
Monitor error rate, latency P99, cost metrics during canary
If canary passes: ./scripts/promote.sh production
If canary fails: ./scripts/rollback.sh production (instant)

14.2 Rollback Runbook
Identify failing version via PagerDuty alert or dashboard
Execute instant rollback: ./scripts/rollback.sh production
Verify health checks pass on previous version
If database migration was applied: ./scripts/migrate.sh rollback --to {previous_migration}
Verify Temporal workflows resumed successfully
Post-incident: create post-mortem, update deployment checklist

14.3 Incident Response Runbook
Acknowledge PagerDuty alert within 5 minutes
Classify severity: P1 (revenue impact), P2 (degraded), P3 (minor), P4 (info)
Open incident Slack channel: #inc-YYYY-MM-DD-{description}
For P1/P2: Page on-call engineer and engineering manager
Diagnose: check Grafana dashboards, Temporal UI, CloudWatch logs
If service-level: restart affected pods, check resource limits
If data-level: quarantine affected records, assess blast radius
If external dependency: activate fallback path, notify affected users
Resolve: deploy fix or rollback
Post-incident review within 48 hours, update runbooks

14.4 Migration Runbook
Review migration SQL: verify reversibility, check for locks on large tables
Test against staging database clone
Schedule during low-traffic window (2-4 AM ET)
Take database snapshot before migration
Apply migration: ./scripts/migrate.sh apply --migration {number}
Verify migration: ./scripts/migrate.sh verify --migration {number}
If verification fails: ./scripts/migrate.sh rollback --migration {number}
Monitor application health for 30 minutes post-migration

14.5 Scaling Runbook
Identify capacity pressure: check autoscaler events, CPU/memory utilization
For predictable load (events, launches): pre-scale 2 hours before
For browser service: increase both pod count and proxy pool size simultaneously
For agent service: increase pod count and verify Temporal worker capacity
For memory service: scale ChromaDB separately from API pods
After peak: allow autoscaler to scale down (respect cooldown periods)

14.6 Key Rotation Runbook
Generate new credential via provider (AWS, Anthropic, social platform)
Store new credential in Secrets Manager as new version
Update service configuration to use new version
Verify service health with new credential
Revoke old credential after 24-hour grace period
Update rotation log with date, rotated key, operator
