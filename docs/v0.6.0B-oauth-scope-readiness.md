## 2. Founder corrections incorporated

### 2.1 Scope classification language

Earlier wording incorrectly stated `calendar.events.readonly` is restricted as fact. Corrected wording (applies anywhere we discuss this scope):

> `calendar.events.readonly` is the narrowest useful Calendar event-read scope for this capability. Google documents it as allowing the app to view events on all calendars the user can access. **Google Cloud Console confirms (2026-06-08) the scope is classified Sensitive for this OAuth project**, description "View events on all your calendars." Sensitive scopes require OAuth verification before going to production (general availability) but do NOT require a CASA security assessment. Compare to `gmail.readonly`, which Cloud Console classifies as Restricted (CASA required at verification time).

This is one of the load-bearing founder facts that lives in §3 below (now answered as F1).

### 2.2 Why `calendar.readonly` is NOT needed now

- `calendar.readonly` is **broader than the first use case**. It includes calendar-list metadata (which calendars exist, sharing/ACL info, color, default reminders).
- `calendar.events.readonly` is **enough for event relevance** (the only signal v0.6.0C/D consume).
- If future Brevio needs full calendar metadata (e.g. distinguishing "Work" vs "Personal" calendars, multi-calendar ranking), it gets its **own scope gate later**.
- **Do not request broader scope now just because long-term Brevio may eventually need it.** Over-broad scope today raises CASA cost AND user-trust risk for zero current value.

### 2.3 Window configurability (not a ceiling)

The 48-hour window is the **first FOMO/email-ranking context window**, not Brevio's long-term production limit.

Long-term Brevio should support intent-based calendar windows:
- urgent email relevance: 24–48 hours
- weekly preparation: 7 days
- travel/events/deadlines: 14–30 days
- broader planning: separately gated, not raw-calendar dumping

**v0.6.0C lock:** `windowHours` MUST be a configurable parameter from day one. Tests MUST prove the service respects a supplied window, not only the default. The 48-hour value is the production default for the first email-ranking use case only.
