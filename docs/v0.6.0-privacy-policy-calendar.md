## Section to add to the existing Brevio privacy policy

Suggested heading: **Google Calendar (read-only)** — placed under whatever section currently covers Gmail access.

> ### Google Calendar (read-only)
>
> Brevio requests read-only access to your Google Calendar through the `https://www.googleapis.com/auth/calendar.events.readonly` scope. This access is used solely to help Brevio understand whether an incoming email is time-sensitive — for example, an email about a meeting that is on your calendar tomorrow morning.
>
> **What Brevio reads.**
> Brevio reads events from your **primary calendar only**, within a rolling **48-hour window** from the current time. Brevio reads only three fields from each event:
>
> - the event title (event summary),
> - the event start time,
> - the event end time.
>
> **What Brevio does NOT read.**
> Brevio never reads, stores, or transmits:
>
> - event attendee names or email addresses,
> - event descriptions, notes, or agendas,
> - event locations or addresses,
> - event attachments,
> - meeting URLs, conference data (Google Meet links, dial-in info), or any links contained in events,
> - any calendar other than your primary calendar,
> - your free/busy availability beyond the 48-hour window above.
>
> **Private events.**
> If an event is marked private (visibility = private) in Google Calendar, Brevio reads only the start and end times. The event title is replaced with the placeholder text "Busy" before it reaches any internal processing. The actual private event title is never read or stored.
>
> **How long Brevio keeps this data.**
> Calendar event data is **ephemeral**. Brevio processes the three approved fields in memory only, for the duration of a single email-ranking decision, and discards them immediately afterward. Calendar event content is **never written to a database, never written to logs, never stored in any persistent storage, never transmitted to any third party, and never used to train any machine learning model**.
>
> **What gets recorded for operational purposes.**
> Brevio's audit log records only structural metadata about Calendar usage — for example, the number of events in the 48-hour window and the time offset to the nearest upcoming event. **The audit log never records event titles, times, attendees, descriptions, or any other event content.**
>
> **No writes.**
> Brevio does not create, modify, move, delete, or otherwise change anything in your Google Calendar. The `calendar.events.readonly` scope is read-only by design, and Brevio does not request any write scope.
>
> **Window flexibility.**
> The 48-hour window above is Brevio's default window for the email-ranking use case. Future Brevio features may introduce additional, separately documented use cases that require different windows (for example, weekly preparation, travel and deadlines, broader planning). Any new use case will be added to this privacy policy before it is enabled, and the relevant access will remain subject to the same field-exclusion and ephemerality rules described above.
>
> **Revoking access.**
> You can revoke Brevio's Google Calendar access at any time at <https://myaccount.google.com/permissions> without affecting your Brevio account or your Gmail access. Brevio's calendar features will simply stop functioning until you re-grant access.

## Why this wording satisfies Google's verification expectations

- **Explicit scope name** (`calendar.events.readonly`) so reviewers can match to the consent screen.
- **Specific use case** ("help Brevio understand whether an incoming email is time-sensitive") — required by Google's restricted-scope policy.
- **Minimum-necessary justification** — three fields read, not all events, primary calendar only.
- **Field-exclusion list** — explicit enumeration of fields NOT read.
- **Ephemerality + no-training assertion** — directly addresses Google's data-handling questionnaire items.
- **No-writes assertion** — closes the "could you use a narrower scope?" reviewer question.
- **Revocation instructions** — required by Google's policy.

## Why this wording matches Brevio's deny-by-default posture

- **Hard list** of excluded fields rather than "we may also access X if needed" hedging.
- **Audit-log boundary explicit**: structural counters only; never event content.
- **Private-event handling** ("Busy" placeholder) named in user-facing terms, matching the deterministic v0.6.0C rule.
- **Window described as a default with future expansion explicitly gated by policy updates** — avoids overclaiming "permanent ceiling" while preserving the 48-hour starting point.

## What is intentionally NOT promised

- No claim of any specific approval status from Google.
- No claim about the scope being "non-sensitive" (per v0.6.0B founder correction: treat as OAuth-sensitive until Cloud Console confirms).
- No promise that future windows will remain at 48h.
- No SLA on processing time.
