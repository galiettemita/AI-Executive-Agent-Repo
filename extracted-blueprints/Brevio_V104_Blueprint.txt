

V10.4 NON-NEGOTIABLE RULES


1. WHY OUTBOUND CALLING IS NEEDED
Many real-world tasks cannot be completed digitally. Restaurants, salons, medical offices, and local service providers often accept reservations by phone only. Brevio must be able to place calls autonomously on behalf of users, conduct a natural conversation, and report back with the outcome — without any user involvement during the call itself.

2. PROVIDER SELECTION
VAPI.ai is the primary provider. Retell AI is the hot-swap fallback. Both are fully specified below so the Codex agent can implement either without additional research.

3. ARCHITECTURE — 3-PLANE INTEGRATION
Outbound calling is implemented in the Executor service (Hands plane) as a new package: hands/call/. Brain proposes and plans. Control approves. Executor executes. Gateway delivers results.

11. WHAT REAL-TIME VOICE MEANS FOR BREVIO
Part B adds a live, bidirectional audio session between the user and Brevio. Put on a headset, open the Brevio web or mobile app, and speak naturally. Brevio listens, processes, and speaks back — sub-second, no messages needed. This is NOT Part A (which places calls to businesses). This is you talking to Brevio directly.

12. PROVIDER SELECTION — LIVEKIT
LiveKit (open-source, self-hostable) is selected for WebRTC infrastructure. The LiveKit Agents framework allows Brevio to run as an AI participant in the audio room — not a relay. Daily.co is the managed fallback.

14. SIGNAL CHAIN — SPEECH TO RESPONSE
End-to-end latency target: <900ms from user stops speaking to Brevio begins speaking. All stages are pipelined — no stage waits for the previous to fully complete.
