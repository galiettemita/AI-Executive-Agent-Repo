1. Executive Summary
BREVIO V9 is the single, authoritative, machine-executable master specification for the BREVIO AI Executive Assistant system. It consolidates four source documents into one deterministic, production-final blueprint that a coding agent can implement end-to-end without discretionary design decisions.
V9 reconciles and supersedes: (1) Blueprint V7, which resolved all conflicts across v6.0–v6.7 and established the four-plane architecture, full DDL, acceptance gates, and compliance matrix; (2) Blueprint V8, which added discovery/onboarding workflows, namespace neutralization, and the Attached Requirements Set (ARS-001–ARS-004); (3) the Auto-Provisioning Engine Deep Research Upgrade Report, which defined six upgrade packages (A–F) for deterministic capability resolution, RBAC/policy/budget gates, durable provisioning super-workflows with compensation, supply-chain verification, deterministic ranking with LLM replay, and drift/health monitoring; and (4) the 10× Improvement Report, which contributed the three-plane reference architecture, OWASP LLM threat mitigations, OpenTelemetry observability patterns, evaluation frameworks, cost models, and migration phasing.

3.1.4 Service Edges and Protocols
3.2 Deterministic Control-Plane Rule
Only the Control service may authorize any side effect (write). Brain proposes plans but never executes. Every tool call must pass through the ExecutionGate PDP, which evaluates OPA policies, autonomy levels, rate limits, budget constraints, and approval requirements before allowing execution. This is enforced architecturally: the executor service only accepts work from Temporal activities initiated by the control plane.
3.3 Two-Phase Commit Protocol
Every write-capable tool must follow the simulate-then-commit protocol:
Phase 1 – Simulate: Execute tool in dry-run mode. No side effects. Record tool_executions(phase=simulate). Validate output against schema. Run semantic verifiers.
Phase 2 – Commit: Prerequisites: simulate_success + execution_gate_allow + approval_if_required. Execute tool for real with idempotency key. Record tool_executions(phase=commit) + trust_receipt + audit_log_entry.
Idempotency scope: workspace_id + tool_key + logical_action_hash. TTL = 30 days. Key format: idem_<base32hex_min16>.
3.4 Reasoning Loop State Machine
The Brain operates a four-state reasoning loop: PLANNER → EXECUTOR → CRITIC → REFLECTOR → TERMINAL.
Deterministic parameters: temperature=0, top_p=1, presence_penalty=0, frequency_penalty=0, max_output_tokens fixed by tier (T0=256, T1=512, T2=1024, T3=2048).
Plan candidates max: T1=1, T2=2, T3=3 (from V7 plan scorer guidance).
Tool schema ordering: lexical by tool_key. Context item ordering: deterministic sort keys.

6. Temporal Workflows
All workflows run exclusively on Temporal. Workflow code must be deterministic and idempotent. Side effects live in activities. Replays are driven by event history.
6.1 Required Workflows
6.2 Provisioning Super-Workflow (Package C)
The provisioning workflow follows this deterministic state progression:
Preflight → CreateRequest → PolicyGate (Package B) → AllocateOrReuseServer → [AuthRequired? → AwaitOAuthCallback → ExchangeTokens] → [VerifyArtifact (Package D)] → DeployServer → FetchToolSchemas → HealthCheck → CommitRegistry → Active
On any failure, compensation executes in exact reverse order of successful steps: disable tools → revoke tokens → deallocate server → emit compensation_completed event.
Compensation is deterministic. Idempotency-Key is required at the API level. Each step has its own idempotency key. Unique constraints prevent duplicate step creation.
6.3 Capability Resolution Workflow (Package A)
Normalize query → Cache check → [cache hit? return cached] → Rule-based matching → [candidates empty AND allow_llm_fallback? → LLM extraction] → Deterministic ranking (Package E) → Cache write → Return.

7. Policy Framework (OPA)
7.1 ExecutionGate Policy Decision Point
The ExecutionGate PDP evaluates every proposed tool call against OPA policies before allowing execution. The decision is recorded in execution_gate_decisions with full policy_checks_json.
Policy inputs include: workspace autonomy level, domain autonomy level, tool risk_level, user role, rate-limit state, budget state, content firewall result, semantic verification result, and workspace-specific blocked_tool_keys.
7.2 Policy Rule Categories
7.3 Provisioning Policy (Package B)
Deterministic decision algorithm (decide_provisioning_v1): checks denied_server_ids → allowed_server_ids → risk_level vs max_allowed → budget (calls/cost/concurrency) → operator review threshold → OAuth owner approval → MCP deploy owner approval → ALLOW.
RBAC hierarchy: owner > admin > delegate > auditor > operator. Only owner/admin can approve OAuth and deploy MCP servers by default. Operator role for admin console review; not assignable by workspace users.

10. Security and Governance
10.1 Threat Model
10.2 Cryptographic Controls
Approval tokens: HMAC-SHA256 signed. CRITICAL TTL=30s, ELEVATED TTL=5min. Nonce-unique. Key version tracked.
OAuth token storage: AES-256-GCM envelope: {ciphertext, nonce, key_version, encrypted_at}. Key material from Secrets Manager.
Key rotation: Dual-key read window ≥ 10min. Background re-encryption job. key_versions table is authoritative.
Webhook signatures: HMAC-SHA256(app_secret, raw_body_bytes). Timing-safe compare. Replay protection: nonce cache TTL=600s.
Audit log integrity: Hash chain HMAC. Append-only table enforced by DB triggers (no UPDATE/DELETE). Separate DB role.
Artifact verification: Ed25519 signature on image digests (sha256). SBOM + vulnerability scan required.
10.3 Data Classification
Every persisted object carries data_class, sensitivity_label, retention_policy_id, and allowed_processors[]. The Content Firewall assigns content_trust levels. The classifier determines processing restrictions.
10.4 GDPR Compliance

Right to erasure via cascade deletes (user → workspace → all scoped data).
Do-not-remember exclusion rules (memory_exclusion_rules).

12.3 Dockerfile Template
All services use the same Dockerfile pattern (Go 1.22, distroless, non-root, read-only FS):
FROM golang:1.22 AS build
ARG SERVICE
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
    go build -trimpath -ldflags="-s -w" -o /out/app ./cmd/${SERVICE}

FROM gcr.io/distroless/static:nonroot
WORKDIR /
COPY --from=build /out/app /app
USER 65532:65532
ENTRYPOINT ["/app"]
Container hardening: distroless base, non-root (65532:65532), read-only filesystem, drop all Linux capabilities, SBOM generation at build, daily CVE scans.
