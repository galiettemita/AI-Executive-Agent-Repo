1. Executive Summary — The Intelligence Ceiling
V9 through V10.2 build an exceptional execution infrastructure. The reasoning loop, memory scaffolding, context engineering, and production hardening are production-grade. V10.2 closed the most urgent prompt and algorithm gaps.

What remains is a deeper class of problem: the cognitive architecture itself. Brevio currently implements a competent linear reasoning pipeline. What it lacks are the cognitive mechanisms that separate reactive task execution from genuine intelligence — the ability to hold beliefs under uncertainty, reason by analogy, model what the user doesn't know, learn implicitly from behavior, think about its own thinking, and anticipate rather than only respond.

V10.3 closes 12 cognitive architecture gaps derived from three sources: (1) dual-process cognitive theory, (2) frontier AI research from 2024–2026 (Graph-of-Thought, self-consistency, constitutional preference learning), and (3) cognitive science research on memory consolidation, prospective memory, and theory of mind.



2. COG-01 — Dual-Process Cognitive Architecture

2.1 Current State & Gap
Brevio's 4-tier reasoning depth (INSTANT → SHALLOW → STANDARD → DEEP) maps structurally to dual-process. But the INSTANT tier is purely cache-based: it hits or misses. If miss, it falls to SHALLOW. There is no intermediate 'learned intuition' layer — a fast pattern recognizer that has internalized thousands of successful executions into implicit heuristics.

Decision Logic: INSTANT Tier (Updated)
function instantTierDecide(request_embedding, workspace_id):
  // 1. Check heuristic layer FIRST (before cache check)
  heuristic = queryHeuristics(request_embedding, workspace_id, threshold=0.88)
  if heuristic AND heuristic.activation_count >= 10:
    return { source: 'HEURISTIC', skill_sequence: heuristic.skill_sequence }
  // 2. Fall back to precomputed cache
  cached = queryFastPathCache(request_embedding, workspace_id)
  if cached AND not expired:
    return { source: 'CACHE', response: cached.answer_payload }
  // 3. Neither — fall to SHALLOW
  return { source: 'MISS' }


3. COG-02 — Graph-of-Thought Reasoning

3.1 Current State & Gap
The PLANNER→EXECUTOR→CRITIC→REFLECTOR loop is linear. Even in DEEP tier, the plan candidates (max 3 for T3) are evaluated sequentially. There is no mechanism for the agent to explore two conflicting sub-strategies in parallel, discover which assumptions held, and synthesize a merged conclusion.

GoT Algorithm
function graphOfThought(problem, max_nodes=20, max_depth=4):
  root = createThought(problem, type='root')
  active_frontier = [root]
  while frontier not empty AND total_nodes < max_nodes:
    for each node in active_frontier:
      branches = generateBranches(node, count=2)  // 2 divergent sub-thoughts
      for branch in branches:
        branch.critic_score = CRITIC.evaluate(branch)
        if branch.critic_score < 0.40: prune(branch)
        else: add_to_frontier(branch)
    if len(active_frontier) >= 2:
      merge_candidates = selectMergePairs(active_frontier)
      for (a, b) in merge_candidates:
        merged = mergThoughts(a, b)
        if merged.critic_score > max(a.score, b.score): promote_merge(merged)
  return concludeGraph(best_frontier_nodes)


4. COG-03 — Metacognitive Monitor

4.1 Current State & Gap
Brevio has domain autonomy levels and trust scores (V9.1). These govern execution permissions. What's missing is the cognitive equivalent: adjusting reasoning depth and confidence thresholds based on domain-specific performance history. An agent that has been wrong 30% of the time on financial planning tasks should automatically escalate those to STANDARD reasoning even if complexity signals suggest SHALLOW.

function metacognitiveAdjust(domain, raw_tier, workspace_id):
  perf = queryDomainPerformance(workspace_id, domain)
  if perf.empirical_success_rate < 0.75:
    floor = perf.metacognitive_tier_floor  // e.g. 'STANDARD'
    if TIER_RANK[raw_tier] < TIER_RANK[floor]:
      emit METACOGNITIVE_UPGRADE(domain, raw_tier, floor)
      return floor
  if perf.empirical_success_rate > 0.95 AND perf.total_executions_30d > 50:
    // Optionally downgrade (save compute on high-confidence domains)
    return max(INSTANT, raw_tier - 1)
  return raw_tier


5. COG-04 — Theory of Mind Layer

5.1 Current State & Gap
USER.md models who the user is. SOUL.md models how Brevio behaves. Neither captures a model of what the user currently knows or believes. If the user asks 'can you book the restaurant for Tuesday?' while their calendar shows a conflict on Tuesday that they clearly haven't noticed, Brevio either books it (wrong) or says 'you have a conflict' (reactive). A ToM-equipped agent proactively surfaces: 'You have a board meeting Tuesday 7-9pm — did you know?' before answering.

The planner decides whether to: (A) surface the gap before executing, (B) adjust the plan to account for it, or (C) note it at the end of the response.


6. COG-05 — Bayesian Belief Updater

6.1 Current State & Gap
V9.1 learned_lessons stores active rules as binary confirmed/denied. A lesson is either active or it isn't. This means preference modeling is: 3 corrections → proposed → confirmed → rule. The rule fires 100% of the time until contradicted. There is no uncertainty, no context-dependence, no gradient. The result is brittle — one context-specific correction overwrites a broadly-accurate rule.

Bayesian Update Rule
function updateBelief(belief, observation_value, observation_weight=1.0):
  // Bayesian update with Normal-Normal conjugate prior
  N = belief.observation_count
  prior_mean = belief.mean
  prior_var  = belief.variance
  // Update mean
  new_mean = (prior_mean / prior_var + observation_value / OBS_VARIANCE) /
             (1 / prior_var + 1 / OBS_VARIANCE)
  // Update variance (decreases as observations accumulate)
  new_var = 1 / (1/prior_var + 1/OBS_VARIANCE)
  // Update belief in DB
  UPDATE belief_distributions SET mean=new_mean, variance=new_var,
    observation_count=N+1, last_updated_at=now()


7. COG-06 — Prospective Memory Engine

7.1 Current State & Gap
Brevio has retrospective memory (what happened), goal tracking (what to achieve), and cron scheduling (what to do at a set time). None of these handle the prospective memory class: 'remember to X when event Y occurs', where Y is a semantic event (not a clock time) and X is a context-dependent action. The difference: a cron fires at 9am every Tuesday. Prospective memory fires the next time the user mentions a specific person, topic, or context.

function checkProspectiveTriggers(request_embedding, request_text, workspace_id):
  // Fast pre-filter: keyword overlap
  candidates = queryByKeywordOverlap(request_text, workspace_id)
  // Semantic match for candidates
  for mem in candidates:
    similarity = cosineSimilarity(request_embedding, mem.trigger_embedding)
    if similarity >= 0.72:
      injectIntoContext(mem)  // surfaced before PLANNER
      if mem.fire_count + 1 >= mem.max_fires: markFired(mem)
      else: incrementFireCount(mem)

10. COG-09 — Active Information Acquisition

10.1 Current State & Gap
Brevio asks clarifying questions when intent confidence < 0.70 (fallback path). The question is generated by the PLANNER as a natural language clarification request. There is no mechanism to evaluate which question to ask, how much uncertainty it would resolve, or whether a question is necessary at all (vs. acting with the best current estimate and course-correcting).

Ask-vs-Act Decision Rule
function askOrAct(intents, clarification_candidates, workspace_id):
  best_question = max(clarification_candidates, key=estimated_gain)
  top_intent_confidence = max(intents, key=confidence).confidence
  // If top intent is close to threshold AND acting-then-correcting is low-risk
  if top_intent_confidence >= 0.60 AND task_is_reversible(top_intent):
    return ACT_WITH_MONITORING  // act on best estimate, watch for correction
  if best_question.estimated_gain < 0.20:
    return ACT_WITH_BEST_ESTIMATE  // question wouldn't help much
  return ASK(best_question)

12. COG-11 — Concept Drift Detector

12.1 Current State & Gap
V10.2 added a KnowledgeFileDriftDetector for SOUL.md/USER.md staleness. This covers intentional preference drift. What's missing is behavioral drift detection: the agent noticing that its predictions are systematically less accurate lately, or that the user's behavior has shifted significantly from their historical baseline. A user who just changed jobs will have entirely different priorities, tools, contacts, and time constraints — the agent's learned patterns from the previous role are now counterproductive.

Drift Detection Algorithm
function detectDrift(workspace_id):
  current = computeWindow(workspace_id, last_14_days)
  baseline = queryCurrentBaseline(workspace_id)  // 90-day rolling average
  // Jensen-Shannon divergence on topic distribution
  topic_drift = JS_divergence(current.topic_dist, baseline.topic_dist)
  skill_drift  = JS_divergence(current.skill_dist, baseline.skill_dist)
  behavior_drift = |current.override_rate - baseline.override_rate|
  drift_score = 0.4*topic_drift + 0.4*skill_drift + 0.2*behavior_drift
  if drift_score > DRIFT_THRESHOLD (default 0.25):
    emit DriftAlert(workspace_id, drift_score, dominant_shift)
    // Agent surfaces: 'I've noticed your focus has shifted lately...'


13. COG-12 — Dynamic Few-Shot Constructor

13.1 Current State & Gap
The PLANNER, CRITIC, REFLECTOR, and PROMPT_RESPONSE_GEN prompts use static system prompts. They have no dynamic few-shot examples. The accumulated history of successful executions — thousands of (request, plan, execution, outcome) tuples — is stored in execution logs but never used to improve the prompts that generate new plans.


13.2 Improvement: Dynamic Example Selection

Few-Shot Example Selection Algorithm
function constructFewShot(request_embedding, prompt_type, workspace_id, K=3):
  // Step 1: retrieve top-20 semantically similar cases
  candidates = queryCaseLibrary(request_embedding, workspace_id, limit=20)
  // Step 2: filter by prompt_type relevance
  candidates = filter(candidates, lambda c: c.domain matches prompt_type)
  // Step 3: MMR selection for diversity
  selected = []
  while len(selected) < K AND len(candidates) > 0:
    if len(selected) == 0:
      selected.append(max(candidates, key=outcome_score))
    else:
      scores = [(c, lambda_sim*sim(c, request) - (1-lambda_sim)*max_sim(c, selected))
                for c in candidates]
      selected.append(max(scores, key=score).candidate)
    candidates.remove(selected[-1])
  return formatAsDemonstrations(selected)

New Column on case_library

Token budget for few-shot examples: 15% of the total token allocation for the prompt type, sourced from the context_budget expansion reserve. MMR lambda (similarity-diversity tradeoff): 0.65 (slightly favor similarity over diversity).

14.4 Closure Statement
BREVIO V10.3 adds 12 cognitive architecture modules derived from cognitive science, neuroscience, and frontier AI research. It adds 11 new tables, extends 2 existing tables, specifies 6 new Temporal workflows, and defines 4 new prompt specifications.

V10.3 is purely additive. It does not modify any prior version's tables, workflows, policies, prompts, or schemas. All prior acceptance gates remain in force.

The cognitive architecture improvements in V10.3 do not improve Brevio for the first interaction. They compound: modules like COG-01, COG-05, COG-08, and COG-10 become meaningfully powerful after 30–90 days of usage. The long-term ceiling on intelligence scales with usage, not just with model improvements. This is the correct design for a personal AI that is meant to grow with its user.

Any behavior not explicitly defined in V9 through V10.3 is forbidden.
